# 바이브코딩 입문 강의 계획서

## 문서 목적

이 문서는 `바이브코딩을 한 번도 해보지 않은 사람`을 대상으로,  
`AI에게 물어보는 단계`에서 `AI에게 일을 시키는 단계`로 넘어가게 만드는 입문 강의의 초안이다.

운영 전제는 `매주 1시간씩 진행되는 연속 강의`다.

최종 방향은 다음과 같다.

1. 처음에는 `Claude Desktop` 환경에서 작은 성공 경험을 만든다.
2. 다음에는 `IDE형 도구`를 통해 더 긴 작업 흐름을 경험하게 한다.
3. 마지막에는 `CLI + WSL` 환경으로 넘어가서 실제 개발 생산성 흐름으로 유도한다.
4. 이 과정 전체에서 `GitHub`, `Node.js`, `Python`, `WSL`의 필요성을 자연스럽게 체감하게 한다.

이 문서는 이후 스크린샷, 예시 프로젝트, 실습 흐름이 추가되면 HTML 발표 자료로 변환하기 쉽게 작성되어 있다.

---

## 1. 강의 핵심 메시지

이 강의의 핵심은 단순히 "AI에게 질문 잘하는 법"을 알려주는 것이 아니다.

핵심은 다음 한 문장으로 정리할 수 있다.

> 바이브코딩은 AI에게 답을 받아보는 경험이 아니라, AI에게 일을 맡기고 결과를 통제하는 경험이다.

초보자는 보통 아래 단계에 머문다.

- 검색창처럼 AI에게 질문한다.
- 코드가 나와도 어디에 넣어야 할지 모른다.
- 프로젝트 구조, 파일, 실행 방법, 수정 흐름을 모른다.
- 결과가 안 되면 다시 처음부터 묻는다.

이번 강의에서는 이 상태에서 다음 상태로 이동시키는 것이 목표다.

- AI에게 작업 목표를 준다.
- 결과를 파일 단위로 받는다.
- GitHub에 올리고 공유한다.
- Node.js / Python으로 실행한다.
- IDE에서 수정한다.
- CLI에서 더 정교하게 다룬다.

---

## 2. 대상 청중 가정

- Git을 잘 모르거나 거의 처음이다.
- GitHub 계정은 있을 수도 있고 없을 수도 있다.
- 개발 환경 설치 경험이 부족하다.
- 코딩은 익숙하지 않지만 AI 서비스는 써본 적이 있다.
- "코드를 직접 짜는 것"보다 "AI와 함께 결과물을 만드는 것"에 관심이 있다.
- Windows 사용자가 대부분이라고 가정한다.

이 문서는 특히 `Windows 사용자에게 WSL 환경으로 진입시키는 것`을 강하게 목표로 둔다.

---

## 3. 강의 목표

수강생이 강의 종료 후 아래를 할 수 있도록 한다.

- 바이브코딩이 무엇인지 자기 말로 설명할 수 있다.
- `질문형 사용`과 `작업지시형 사용`의 차이를 이해한다.
- GitHub가 왜 필요한지 이해한다.
- Node.js와 Python이 왜 필요한지 이해한다.
- Windows에서는 왜 WSL이 중요한지 이해한다.
- 데스크톱 AI 도구, IDE형 도구, CLI형 도구의 차이를 이해한다.
- 아주 작은 프로젝트를 AI와 함께 시작하고 실행해볼 수 있다.

### 필수 개념 장착

이 강의는 도구 사용법만 다루면 안 된다.  
초보자라도 아래 개념은 반드시 장착하고 가는 편이 좋다.

- 프론트엔드: 사용자가 보는 화면
- 백엔드: 화면 뒤에서 처리하는 로직
- 데이터베이스: 데이터를 저장하는 장소
- Git: 변경 이력을 관리하는 도구
- GitHub: 프로젝트를 쌓고 공유하는 온라인 작업 공간

핵심은 깊은 이론이 아니라 `역할 구분`이다.

- "이건 화면 문제인가?"
- "이건 데이터 저장 문제인가?"
- "이건 서버 처리 문제인가?"
- "이건 버전 관리 문제인가?"

이 구분이 되어야 AI에게 더 정확하게 일을 시킬 수 있다.

### 운영 전제

- 매주 60분 수업으로 진행한다.
- 전체 과정은 `4주 또는 5주` 구성으로 설계한다.
- 설치는 강의 전에 안내 페이지로 미리 준비하게 한다.
- 강의 시간은 설치보다 `체험`, `실행`, `수정`, `비교`에 집중한다.
- 공통 기반은 `GitHub`, `Node.js`, `Python`, `WSL`이다.
- IDE와 AI 도구는 하나로 강제하지 않는다.
- 시연은 `VS Code + GitHub Copilot`를 기본 예시로 삼되, 수강생은 `VS Code`, `Cursor`, `Antigravity` 등 자신이 쓰는 도구를 사용할 수 있게 한다.
- CLI 단계에서는 `Claude Code`, `GitHub Copilot CLI` 등 대표 도구를 예시로 보여준다.
- 중간 산출물은 우선 `GitHub 저장소 링크`를 기준으로 받는다.
- 사이트 배포는 후반 또는 다음 단계에서 `Vercel`로 확장한다.

## 4. 권장 학습 단계

### 단계 1. Claude Desktop 체험

목표는 `복잡한 설치 없이도, 내가 AI와 함께 뭔가를 만들 수 있다`는 감각을 주는 것이다.

- Claude Desktop을 공통 도구로 활용한다.
- 아주 작은 결과물을 만든다.
- 예시:
  - 자기소개 페이지
  - 간단한 체크리스트 앱
  - 오늘 할 일 정리 페이지
  - 버튼 누르면 문구가 바뀌는 미니 웹페이지

이 단계의 메시지:

- 코딩은 "처음부터 다 직접 쓰는 일"이 아니다.
- AI와 함께 바로 만들 수 있다.
- 하지만 결과물을 계속 다루려면 파일, 버전, 저장소 개념이 필요하다.

#### 1주차 공통 도구 추천안

강의 목적이 `노코드 체험`이 아니라 `바이브코딩 입문`이므로,  
1주차는 모두 같은 대화형 도구를 쓰고, 2주차부터 IDE와 파일 구조로 넘어가는 편이 운영상 자연스럽다.

##### 1순위. Claude Desktop

- 이유:
  - 모든 수강생을 같은 출발점에 세우기 쉽다.
  - 질문형 사용에서 작업지시형 사용으로 넘어가는 연습에 집중하기 좋다.
  - 추후 GitHub, IDE, Claude Code로 넘어갈 때 역할 구분을 설명하기 좋다.
- 강의용 판단:
  - 초보자가 "AI에게 일을 맡긴다"는 감각을 얻기 좋다.
  - 다만 실제 실행 환경은 아니므로, 1주차 산출물을 2주차에 로컬 파일로 옮기는 설계가 함께 필요하다.

##### 보조 후보. Replit

- 이유:
  - 브라우저에서 바로 실행 결과를 보여주기 쉽다.
  - 코드와 미리보기를 같이 설명하기 좋다.
- 강의용 판단:
  - 수업 밖 추가 실험용으로는 좋다.
  - 하지만 1주차 공통 흐름을 통일하는 메인 도구로 두기에는 불필요한 분기가 생긴다.

##### 보조 후보. Bolt.new

- 이유:
  - 브라우저에서 빠르게 결과 화면을 만들기 쉽다.
- 강의용 판단:
  - 보조 데모로는 쓸 수 있다.
  - 하지만 1주차 공통 기준은 Claude Desktop이 더 단순하다.

##### UI 시연용 보조 후보. v0

- 이유:
  - UI/랜딩페이지 시연에는 강하다.
- 강의용 판단:
  - 1주차 공통 도구보다는 보조 시연용이 적절하다.

#### 1주차 도구 최종 추천

현재 강의 목적 기준으로는 아래처럼 추천한다.

- 메인 추천: `Claude Desktop`
- 보조 데모: `Replit` 또는 `Bolt.new`
- UI 시연용 보조 후보: `v0`

#### 1주차 첫 프로젝트 추천

`스타트업 + MBA 수강생` 기준으로는 단순 TODO 앱보다 `의사결정에 바로 닿는 데이터 앱`이 더 적합하다.

가장 추천하는 첫 프로젝트는 아래다.

##### 추천 1순위. 출장 / 미팅 의사결정 대시보드

- 입력:
  - 도시
  - 국가
  - 출장 날짜
- 출력:
  - 날씨
  - 강수 확률
  - 공휴일 여부
  - 환율
  - 간단한 요약 문장
- 왜 좋은가:
  - MBA 수강생에게 바로 맥락이 있다.
  - UI가 복잡하지 않다.
  - `Open-Meteo`, `Nager.Date`, `Frankfurter`를 조합하면 API 키 없이도 만들 수 있다.
  - 나중에 `출장 예산 계산`, `비즈니스 데이 계산`, `도시 비교`로 쉽게 확장 가능하다.

##### 추천 2순위. 글로벌 시장 체크 대시보드

- 입력:
  - 국가 1개 또는 2개
- 출력:
  - 공휴일
  - 환율
  - 기본 거시 지표
- 왜 좋은가:
  - 스타트업, MBA 과정과 맞닿아 있다.
  - `World Bank Indicators API`를 붙이면 국가 비교형 미니 대시보드로 확장할 수 있다.
- 주의점:
  - 첫 주차 프로젝트로는 약간 추상적일 수 있다.
  - 그래서 첫 주에는 출장/미팅 대시보드가 더 쉽다.

##### 비추천. 주차장 정보 앱

- 이유:
  - 한국 주차장 Open API는 보통 인증키가 필요하다.
  - 초반 강의 목표인 `키 없이 바로 성공 경험 만들기`와 맞지 않는다.
  - 지역별 데이터 형식과 품질 차이도 커서 초보자 실습용으로 불리하다.

#### API 후보 탐색 결과 요약

초반 실습용 기준은 아래처럼 정리할 수 있다.

##### 바로 쓰기 좋은 API

- `Open-Meteo`
  - 날씨, 기온, 강수확률, 공기질 등
  - API 키 없이 시작 가능
- `Nager.Date`
  - 국가별 공휴일
  - 브라우저 호출 가능, rate limit 없음
- `Frankfurter`
  - 환율 데이터
  - API 키와 사용량 제한 없이 시작 가능
- `World Bank Indicators API`
  - 국가별 경제/개발 지표
  - 인증 없이 접근 가능

##### 초반 실습용으로 덜 적합한 데이터

- 한국 주차장 정보
  - 공공데이터포털 주차장 API는 `serviceKey` 필수
- 서울 열린데이터광장 Open API
  - 기본적으로 인증키 발급 절차가 필요

#### 초반 실습용 권장 API 조합

가장 현실적인 조합은 아래다.

1. `Open-Meteo` + `Nager.Date`
2. `Open-Meteo` + `Frankfurter`
3. `Open-Meteo` + `Nager.Date` + `Frankfurter`

이 조합이면 `API 키 발급`, `쿼터`, `승인 대기` 없이 바로 시작할 수 있다.

#### 공식 참고 링크

- Open-Meteo Docs: `https://open-meteo.com/en/docs`
- Nager.Date API: `https://date.nager.at/Api`
- Frankfurter Docs: `https://frankfurter.dev/docs`
- World Bank Indicators API: `https://datahelpdesk.worldbank.org/knowledgebase/articles/889392-about-the-indicators-api-documentation`
- 서울 열린데이터광장 FAQ: `https://data.seoul.go.kr/together/notice/faqList.do`
- 공공데이터포털: `https://www.data.go.kr`

### 단계 2. IDE형 체험

목표는 `AI에게 프로젝트 단위로 일을 시키는 감각`을 주는 것이다.

- VS Code + GitHub Copilot을 기본 예시로 소개하되, 수강생은 자신이 쓰는 IDE와 AI 도구를 활용할 수 있게 한다.
- 폴더 단위 작업, 파일 수정, 여러 파일 생성, 실행, 오류 수정 흐름을 경험하게 한다.
- 예시:
  - 기존 HTML 페이지에 섹션 추가하기
  - CSS 스타일 전체 바꾸기
  - FAQ 섹션 추가하기
  - 작은 기능 1개 넣기

이 단계의 메시지:

- 이제 AI는 답변 도구가 아니라 작업 파트너가 된다.
- 파일 단위 수정, 프로젝트 맥락 이해, 반복 개선이 가능해진다.
- 이 시점부터 GitHub 연결과 버전 관리가 사실상 필수다.

### 단계 3. CLI + WSL 체험

목표는 `실제 개발 생산성 환경`으로 들어가게 만드는 것이다.

- 터미널에서 프로젝트를 열고 실행한다.
- Claude Code 또는 GitHub Copilot CLI 같은 도구를 활용한 작업 흐름을 체험한다.
- WSL 환경에서 Node.js, Python, Git을 사용해본다.
- 예시:
  - 저장소 클론
  - 패키지 설치
  - 개발 서버 실행
  - README 보고 프로젝트 실행
  - AI에게 버그 수정 지시

이 단계의 메시지:

- 진짜 일은 결국 로컬 실행 환경에서 이루어진다.
- AI에게 더 큰 일을 시키려면 파일시스템, 실행 환경, 패키지 관리자, 터미널을 다룰 줄 알아야 한다.
- Windows 사용자는 WSL로 들어오는 것이 장기적으로 훨씬 유리하다.

## 5. 왜 GitHub가 필수인가

초보자에게 GitHub는 단순한 개발자 SNS가 아니라 `작업의 본진`이라고 설명하는 것이 좋다.

### 핵심 이유

- AI가 만든 결과물을 저장할 곳이 필요하다.
- 이전 버전으로 돌아갈 수 있어야 한다.
- 다른 사람과 공유할 수 있어야 한다.
- 다른 컴퓨터에서도 이어서 작업할 수 있어야 한다.
- IDE, 배포 도구, AI 코딩 도구 대부분이 GitHub 중심으로 연결된다.

### 초보자에게 이렇게 설명하면 좋다

- GitHub는 `내 프로젝트 폴더를 안전하게 보관하고 기록하는 온라인 작업실`이다.
- 파일을 그냥 바탕화면에 두는 것과, 프로젝트를 GitHub에 올려두는 것은 완전히 다르다.
- 바이브코딩에서 중요한 것은 "잘 물어보는 것"만이 아니라 "계속 이어서 만드는 것"인데, 그 연결고리가 GitHub다.

### 강의용 한 줄 설명

> GitHub가 없으면 AI가 만들어준 결과물이 흩어지고, GitHub가 있으면 프로젝트가 쌓인다.

## 6. 왜 Git이 필요한가

초보자에게 Git은 어렵게 느껴지기 쉽다.  
하지만 바이브코딩 관점에서 Git은 `프로젝트 변경 이력을 안전하게 다루는 최소 도구`로 설명하는 것이 좋다.

### 핵심 이유

- 작업 전후 차이를 기록할 수 있다.
- 실수했을 때 이전 상태로 돌아갈 수 있다.
- 여러 번 수정하면서도 프로젝트를 망가뜨리지 않고 관리할 수 있다.
- GitHub에 올릴 때 변경 내용을 구조적으로 보낼 수 있다.
- AI가 여러 파일을 바꿨을 때 무엇이 달라졌는지 확인하기 좋다.

### 초보자 설명 문장

- Git은 "저장"보다 더 강한 개념이다.
- 단순히 파일 하나를 덮어쓰는 것이 아니라, 어떤 변화가 있었는지 기록해주는 도구다.
- 바이브코딩에서는 AI가 만든 수정 내용을 내가 검토하고 되돌릴 수 있어야 하는데, 그때 Git이 필요하다.

### Git과 GitHub 차이 설명

- Git은 `내 변경 이력을 관리하는 도구`다.
- GitHub는 `그 기록과 프로젝트를 온라인에 저장하고 공유하는 공간`이다.

### 강의용 한 줄 설명

> Git이 작업 기록을 남기고, GitHub가 그 기록을 쌓아두는 장소가 된다.

## 7. 왜 Node.js가 필요한가

초보자는 보통 "왜 갑자기 Node를 깔아야 하죠?"에서 막힌다.  
그래서 Node.js를 `웹 프로젝트를 실행하는 기본 엔진`으로 설명하는 것이 좋다.

### 핵심 이유

- 많은 웹 프로젝트가 Node.js 기반으로 실행된다.
- 패키지 설치 도구(`npm`, `pnpm`, `yarn`)를 쓰기 위해 필요하다.
- React, Next.js, Vite 같은 현대적인 프론트엔드 환경이 거의 다 Node.js를 전제로 한다.
- AI가 만들어준 프로젝트를 로컬에서 실행할 때 가장 자주 필요하다.

### 초보자 설명 문장

- Node.js는 웹 앱을 내 컴퓨터에서 실행하고 관리하는 기본 도구다.
- AI가 "패키지를 설치하고 서버를 실행하세요"라고 말할 때, 그 뒤에는 대부분 Node.js가 있다.

### 강의용 한 줄 설명

> 브라우저에서 보이는 결과물 뒤에는, 대개 Node.js 기반 실행 환경이 있다.

## 8. 왜 Python이 필요한가

초보자에게 Python은 "개발자만 쓰는 언어"가 아니라 `AI 자동화와 도구 연결의 범용 엔진`으로 설명하는 편이 효과적이다.

### 핵심 이유

- AI, 자동화, 데이터 처리, 스크립트 작업에 가장 널리 쓰인다.
- 파일 정리, 변환, 크롤링, 간단한 자동화에서 활용 범위가 넓다.
- 많은 AI 관련 예제, 라이브러리, 실습이 Python 기반이다.
- 나중에 "한 번에 여러 파일 처리", "데이터 정리", "자동 리포트 생성" 같은 작업으로 자연스럽게 확장된다.

### 초보자 설명 문장

- Python은 "조용히 뒤에서 일을 처리해주는 언어"라고 설명할 수 있다.
- 웹 화면만 만드는 것이 아니라, 실제로 일을 자동으로 돌리는 경험으로 가려면 Python이 매우 유용하다.

### 강의용 한 줄 설명

> Node.js가 웹 실행의 기본 엔진이라면, Python은 AI 자동화의 만능 공구함이다.

## 9. 왜 WSL로 유도해야 하는가

Windows 사용자에게는 이 부분이 매우 중요하다.  
이 강의는 `가능하면 초반부터 WSL을 권장하는 흐름`으로 설계하는 것이 좋다.

### WSL을 권장하는 이유

- 실제 개발 환경이 Linux 기준인 경우가 많다.
- 많은 오픈소스 문서와 설치 가이드가 Linux/macOS 기준으로 작성되어 있다.
- 패키지 설치, CLI 도구 사용, 권한 처리, 경로 처리에서 WSL이 더 안정적이다.
- AI가 제안하는 명령어도 Linux 기준일 때가 많아 따라가기가 쉽다.
- Docker, 서버, 배포, 백엔드, 데이터 작업으로 갈수록 Linux 감각이 중요해진다.

### 초보자에게 설명하는 방식

- WSL은 Windows 안에서 Linux 개발 환경을 쓰게 해주는 도구다.
- Windows만으로도 할 수는 있지만, 앞으로 갈수록 자주 부딪히는 문제를 줄이려면 WSL이 훨씬 낫다.
- 즉, 처음에는 조금 낯설어도 나중에 훨씬 덜 고생하는 길이다.

### 강의용 한 줄 설명

> Windows에서 바이브코딩을 오래 하려면, 결국 WSL이 생산성을 지켜준다.

## 10. 필수 개념 설명 방식

초보자에게는 아래처럼 `기능 중심 비유`로 설명하는 것이 좋다.

### 프론트엔드

- 사용자가 직접 보는 화면
- 버튼, 입력창, 카드, 차트, 레이아웃 같은 영역

강의용 설명:

> 프론트엔드는 사용자가 만지는 부분이다.

### 백엔드

- 화면 뒤에서 데이터를 처리하는 영역
- API 호출, 계산, 권한 처리, 데이터 가공 같은 작업

강의용 설명:

> 백엔드는 화면 뒤에서 일을 처리하는 부분이다.

### 데이터베이스

- 데이터를 저장해두는 장소
- 사용자 정보, 일정, 메모, 주문, 기록 등이 들어간다

강의용 설명:

> 데이터베이스는 프로젝트의 기억 창고다.

### Git

- 파일 변경 이력을 관리하는 도구
- 잘못 수정했을 때 돌아가고, 무엇이 바뀌었는지 비교할 수 있다

강의용 설명:

> Git은 프로젝트의 되감기와 기록 기능이다.

### GitHub

- Git 기록과 프로젝트를 온라인에 보관하고 공유하는 공간
- 수업에서는 제출, 공유, 피드백의 중심이 된다

강의용 설명:

> GitHub는 프로젝트가 쌓이는 온라인 작업실이다.

### 초보자에게 꼭 심어줄 문장

> 바이브코딩은 코드를 전부 외우는 일이 아니라,  
> 지금 문제가 화면 문제인지, 로직 문제인지, 데이터 문제인지, 버전 문제인지 구분하는 일에 가깝다.

## 11. GitHub 운영 방식 제안

수업 운영 관점에서는 `GitHub Organization`을 하나 만드는 것이 가장 현실적이다.

### 추천 운영안

- 강의용 GitHub Organization 생성
- 수강생을 Organization 멤버로 초대
- 주차별 저장소 또는 과제용 저장소 운영
- 중간 제출물은 GitHub 저장소 링크로 받기

### 가능한가?

가능하다.

GitHub 공식 문서 기준으로:

- Organization에는 멤버를 초대할 수 있다.
- Team을 만들어 저장소 접근 권한을 줄 수 있다.
- Organization 저장소에 대해 `Read`, `Write`, `Admin` 등 권한을 다르게 부여할 수 있다.

즉, `모두를 초대하고 코드를 보게 하는 것` 자체는 충분히 가능하다.

### 운영 모델 1. 가장 단순한 방식

- 모든 학생이 `public repository`를 사용
- 장점:
  - 설정이 가장 쉽다
  - 링크 공유와 코드 보기 쉬움
- 단점:
  - 수업 외부에도 코드가 공개됨

### 운영 모델 2. 추천 방식

- 강의용 Organization 생성
- 학생별 또는 팀별 `private repository` 생성
- 필요한 경우 Team 권한으로 서로 읽을 수 있게 설정
- 장점:
  - 수업 안에서 관리하기 좋다
  - 공개 범위를 통제할 수 있다
  - 권한 관리가 유연하다

### 운영 모델 3. 교육용으로 가장 정교한 방식

- `GitHub Classroom` 사용
- 장점:
  - 과제 배포와 제출 흐름이 명확하다
  - 교육용 운영에 적합하다
  - 필요하면 Codespaces 같은 브라우저 기반 개발 환경도 연결 가능하다
- 단점:
  - 초반 셋업이 조금 더 필요하다

### 어떤 모델이 가장 적합한가

현재 강의 성격이라면 아래 순서를 추천한다.

1. 처음에는 `Organization + 저장소 링크 제출`
2. 과정이 반복되거나 학생 수가 많아지면 `GitHub Classroom`

### "워크플레이스처럼" 쓸 수 있나?

완전히 Slack 같은 협업 공간은 아니지만,  
수업 관점에서는 `GitHub Organization`이 사실상 워크플레이스 역할을 한다고 보면 된다.

할 수 있는 것:

- 멤버 초대
- 팀 구성
- 저장소별 권한 제어
- 코드 열람
- 이슈와 Pull Request로 피드백

### 코드 열람은 어디까지 가능한가

- public repository면 누구나 열람 가능
- private repository면 권한이 있는 사람만 열람 가능
- Team에 `Read` 권한을 주면 학생들에게 읽기 권한만 줄 수도 있다

즉, `학생 전원이 서로의 코드를 보기`도 가능하고,  
`강사만 모든 코드를 보고 학생끼리는 못 보게 하는 구조`도 가능하다.

### 강의용 추천 결론

- 소규모면: `Organization + private repo + 강사 전체 열람`
- 서로 코드 리뷰도 시키고 싶으면: `Organization + team read access`
- 과제 운영을 체계화하려면: `GitHub Classroom`

## 12. 설치 안내 전략

설치 안내는 단순히 "이거 누르고 설치하세요"가 아니라,  
`왜 이걸 설치하는지`와 `언제 쓰게 되는지`를 같이 설명해야 이탈이 줄어든다.

권장 순서는 다음과 같다.

1. GitHub 계정 만들기
2. Git 설치
3. Node.js 설치
4. Python 설치
5. Windows 사용자는 WSL 설치
6. VS Code 설치
7. GitHub Copilot 설정 확인

### 설치 설명 원칙

- 설치 전에 `왜 필요한지`를 먼저 설명한다.
- 설치 후 `정상 설치 확인 방법`을 바로 보여준다.
- 설치가 끝나면 `어디에 쓰는지`를 바로 연결한다.
- 가능하면 한 번에 다 설치시키지 말고, 실습 직전 필요한 것을 연결해서 설치시킨다.
- 강의 시작 전용 안내 페이지를 만들어 미리 설치하게 한다.

## 13. 설치 안내 방법

설치 안내는 `문서만 던지는 방식`보다 `사전 안내 + 현장 확인 + 실행 검증`의 3단계로 운영하는 것이 좋다.

### 방식 A. 강의 전 사전 안내

- 강의 하루 전 또는 며칠 전에 준비 문서를 미리 보낸다.
- GitHub 계정 생성, Git 설치, Node.js 설치, Python 설치, WSL 설치를 미리 요청한다.
- 설치가 끝난 사람은 체크리스트를 통해 자가 점검하게 한다.

장점:

- 강의 당일 설치 이슈로 시간을 덜 소모한다.
- 본 강의에서 바이브코딩 흐름에 더 집중할 수 있다.

주의점:

- 초보자는 설치 문서만 보고 중간에 멈추기 쉽다.
- 그래서 `설치 확인 명령어`와 `막히는 지점 FAQ`를 같이 줘야 한다.

### 방식 B. 강의 중 함께 설치

- 핵심 도구만 직접 함께 설치한다.
- 설치 직후 바로 확인 명령어를 실행하게 한다.
- 확인이 끝나면 즉시 실습으로 연결한다.

장점:

- 따라오지 못한 사람을 현장에서 구조할 수 있다.
- "왜 이게 필요한지"를 실시간으로 설득할 수 있다.

주의점:

- 설치 시간이 길어지면 강의 몰입이 떨어진다.
- 네트워크, OS 차이, 권한 문제로 예상보다 오래 걸릴 수 있다.

### 권장 운영안

- GitHub 계정 생성은 사전 안내로 보낸다.
- Git / Node.js / Python / WSL 설치는 사전 안내 문서를 제공하되, 강의에서 핵심 확인만 다시 한다.
- 강의 중에는 `설치`보다 `설치 확인과 첫 실행 경험`에 집중한다.
- VS Code와 GitHub Copilot 로그인 확인까지 사전 준비 항목에 포함한다.

### 설치 안내 자료 구성 권장안

- 한 페이지 요약본: 왜 설치하는지, 어디서 받는지, 설치 후 확인 명령어
- 상세 가이드: 운영체제별 설치 절차와 자주 막히는 문제
- 현장 슬라이드: 설치 완료 여부를 빠르게 점검하는 체크 화면
- 강의용 안내 페이지: 수강생이 강의 전 직접 따라할 수 있는 링크 모음과 체크리스트

## 14. How To Install 초안

아래는 강의 자료에 바로 넣을 수 있는 설치 안내 초안이다.

### 14-1. GitHub 계정 만들기

목적:

- 프로젝트를 저장하고 공유하기 위해 필요하다.
- 나중에 AI 도구, IDE, 배포 서비스와 연결된다.

안내 문구 초안:

1. GitHub 사이트에 접속한다.
2. 개인 계정을 만든다.
3. 이메일 인증을 완료한다.
4. 프로필 이름과 사용자명을 정한다.

실습 연결:

- 첫 저장소를 만들고 프로젝트를 올리는 단계로 이어진다.

### 14-2. Git 설치

목적:

- GitHub와 내 컴퓨터를 연결하는 버전 관리 도구다.

안내 문구 초안:

1. Git 공식 사이트에서 설치 파일을 받는다.
2. 기본 옵션으로 설치한다.
3. 설치 후 터미널에서 `git --version` 으로 확인한다.

초보자에게 강조할 점:

- Git은 "과거 버전으로 돌아가기"와 "작업 기록 남기기"를 가능하게 해준다.
- GitHub를 제대로 쓰려면 결국 Git이 필요하다.

### 14-3. Node.js 설치

목적:

- 웹 프로젝트 실행과 패키지 설치를 위해 필요하다.

안내 문구 초안:

1. Node.js 공식 사이트에서 `LTS 버전`을 설치한다.
2. 설치 후 터미널에서 `node -v` 와 `npm -v` 를 확인한다.

초보자에게 강조할 점:

- React, Next.js, Vite 같은 현대 웹 프로젝트는 대부분 Node.js가 필요하다.
- AI가 만들어준 프론트엔드 프로젝트를 실행하려면 거의 필수다.

### 14-4. Python 설치

목적:

- AI 자동화, 스크립트, 데이터 처리, 간단한 도구 제작에 필요하다.

안내 문구 초안:

1. Python 공식 사이트에서 최신 안정 버전을 설치한다.
2. 설치 과정에서 `Add Python to PATH` 옵션을 체크한다.
3. 설치 후 `python --version` 또는 `python3 --version` 으로 확인한다.

초보자에게 강조할 점:

- Python은 "웹 화면"이 아니라 "뒤에서 일하게 만드는 자동화"로 이어진다.
- 앞으로 AI를 더 강하게 활용할수록 Python 활용 폭이 커진다.

### 14-5. WSL 설치

대상:

- Windows 사용자

목적:

- Linux 기반 개발 환경을 쉽게 쓰기 위해 필요하다.

안내 문구 초안:

1. Windows 터미널 또는 PowerShell을 관리자 권한으로 연다.
2. `wsl --install` 명령으로 설치를 진행한다.
3. 재부팅 후 Ubuntu 같은 배포판 초기 설정을 진행한다.
4. WSL 터미널에서 개발 작업을 시작한다.

설치 후 확인:

- `wsl -l -v`
- WSL 안에서 `git --version`
- WSL 안에서 `node -v`
- WSL 안에서 `python3 --version`

강조 메시지:

- 앞으로 CLI 기반 AI 도구를 쓸 때 WSL 쪽이 훨씬 덜 막힌다.
- Windows 파일 탐색기보다 `WSL 홈 디렉토리 안에서 프로젝트를 작업하는 습관`을 만드는 것이 중요하다.

### 14-6. VS Code 설치

목적:

- IDE형 바이브코딩 경험을 위한 대표 도구다.

안내 문구 초안:

1. VS Code를 설치한다.
2. GitHub 로그인 연결을 한다.
3. GitHub Copilot 확장을 설치한다.
4. 폴더를 열고 AI와 함께 파일 단위 수정을 시작한다.

보완 메시지:

- VS Code는 `기본 예시`다.
- 수강생이 이미 Cursor, Antigravity 등 다른 도구를 쓰고 있다면 그 도구를 사용해도 된다.
- 중요한 것은 도구 브랜드가 아니라 `파일 편집`, `프로젝트 실행`, `GitHub 연결`이 가능한가다.

### 14-7. GitHub Copilot 설정 확인

목적:

- "질문형"에서 "작업지시형"으로 넘어가는 핵심 경험을 만든다.

안내 문구 초안:

- "이 기능을 추가해줘"
- "이 파일 구조를 유지하면서 디자인만 바꿔줘"
- "오류를 분석하고 수정해줘"
- "README를 보고 실행 가능한 상태까지 만들어줘"

초보자에게 강조할 점:

- 중요한 것은 정답 하나를 받는 것이 아니라, 작업을 분해해서 AI에게 맡기는 것이다.
- 학생 대상이라면 GitHub Education 인증과 Copilot 사용 가능 여부를 강의 전에 확인하게 한다.

보완 메시지:

- Copilot은 대표 예시일 뿐 필수 단일 선택지는 아니다.
- 수강생이 OpenAI, Google AI, Claude, Copilot 기반 도구를 이미 사용 중이라면 그대로 활용하게 할 수 있다.
- 단, 강의 안내 문서에는 `최소 1개의 AI 작업 도구를 로그인 가능한 상태로 준비`하도록 쓰는 편이 현실적이다.

## 15. 추천 강의 흐름 초안

### 오프닝

- 코딩을 잘 몰라도 AI와 함께 결과물을 만들 수 있다는 점을 보여준다.
- 하지만 단순 질문만으로는 프로젝트가 쌓이지 않는다는 점을 보여준다.

### 파트 1. 바이브코딩이란 무엇인가

- 질문형 AI 사용과 작업지시형 AI 사용의 차이 설명
- "답변 받기"와 "일 시키기"의 차이 설명
- 결과물 중심 사고 소개

### 파트 2. Claude Desktop 작은 성공 경험

- Claude Desktop에서 바로 되는 아주 작은 프로젝트 시연
- 빠르게 결과물을 보고 성취감 형성

### 파트 3. 왜 GitHub / Node.js / Python이 필요한가

- 프로젝트를 이어가기 위한 필수 기반 설명
- 설치가 귀찮은 것이 아니라, 확장을 위한 기반이라는 점을 설득

### 파트 4. IDE형 경험

- VS Code + GitHub Copilot 시연
- 여러 파일을 다루는 흐름 소개
- AI와 협업하는 감각 소개

### 파트 5. WSL + CLI로 넘어가기

- 왜 결국 로컬 실행 환경과 터미널이 필요한지 설명
- Windows 사용자에게 WSL을 권장하는 이유 설명
- Claude Code 기반의 아주 기본적인 흐름 소개

### 파트 6. 마무리

- 지금 당장 할 수 있는 첫 실습 제안
- 설치 체크리스트 제공
- 다음 단계 로드맵 제공
